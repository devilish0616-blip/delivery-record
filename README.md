# 員工每日送件紀錄系統

純前端網頁應用程式，以 Google Sheets 作為資料庫，透過 Google OAuth 2.0 讀寫資料，
不需要自架後端伺服器。所有功能皆包含在單一檔案 `index.html` 中。

## 功能簡介

### 📋 每日紀錄（所有人皆可使用）
- 記錄日期、員工姓名、車牌號碼、結束里程、使用公里數（自動計算）、正物流件數、逆物流件數、備註
- 起始里程會自動帶入該車牌「上次紀錄的結束里程」，使用者不需手動輸入
  - 若該車牌尚無歷史紀錄，起始里程視為與結束里程相同（使用公里數 = 0）
- 每月自動建立對應的工作表（格式：YYYY-MM）
- 換機油提醒：當行駛里程超過設定值（預設 1000km）會跳出提醒視窗，可選擇繼續提交或取消

### ⚙️ 系統管理（限管理員 Google 帳號）
- **數據儀表板**：本月總送件數、各員工送件數排行、各車牌使用公里數統計
- **員工管理**：新增/刪除員工、編輯職稱
- **車牌管理**：新增/刪除車牌
- **換機油紀錄管理**：設定/更新各車牌上次換機油里程與日期，並顯示是否需要換機油
- **薪水計算**：功能預留，尚未開放

## 一、設定步驟

### 1. 設定檔（index.html 開頭的「設定區塊」）

```js
const SPREADSHEET_ID  = '你的 Google Sheets 試算表 ID';
const OAUTH_CLIENT_ID = '你的 OAuth 2.0 用戶端 ID';
const OIL_CHANGE_INTERVAL_KM = 1000; // 換機油提醒：行駛超過此公里數即提醒

// 允許進入「系統管理」後台的 Google 帳號 Email 清單
const ADMIN_EMAILS = [
  'your-gmail@gmail.com'
];
```

- `SPREADSHEET_ID`：Google 試算表網址中 `/d/` 與 `/edit` 之間的字串
- `OAUTH_CLIENT_ID`：在 Google Cloud Console 建立的 OAuth 2.0 用戶端 ID
- `ADMIN_EMAILS`：可填入多組 Email（以逗號分隔多筆），只有清單內的帳號登入後才能使用「系統管理」分頁

### 2. Google Cloud Console 設定

1. 建立專案，並啟用「**Google Sheets API**」
2. 建立「**OAuth 用戶端 ID**」（應用程式類型：網頁應用程式）
3. 在「**已授權的 JavaScript 來源**」加入網站網址（例如 `https://your-username.github.io`，僅網域、不含路徑）
4. OAuth 同意畫面設定：
   - 新增範圍：`.../auth/spreadsheets`、`.../auth/userinfo.email`
   - 若狀態為「測試中」，需將會使用本系統的所有 Google 帳號加入「測試使用者」名單

### 3. Google 試算表設定

#### 「系統設定」工作表（A1 標題列：類型 / 名稱 / 職稱）

| 類型 | 名稱     | 職稱   |
|------|----------|--------|
| 員工 | 王小明   | 司機   |
| 車牌 | ABC-1234 |        |

- A 欄填入「員工」或「車牌」
- 「員工」列的 B 欄填姓名、C 欄填職稱
- 「車牌」列的 B 欄填車牌號碼，C 欄留空

#### 「換機油紀錄」工作表（A1 標題列：車牌號碼 / 上次換機油里程 / 換機油日期 / 備註）

| 車牌號碼 | 上次換機油里程 | 換機油日期 | 備註 |
|----------|----------------|------------|------|
| ABC-1234 | 12000          | 2026-05-01 |      |

> 以上兩張工作表若不存在，系統會在第一次登入時自動建立並寫入標題列。
> 但建議先手動建立並輸入初始資料，避免員工第一次提交紀錄時查無車牌資訊。

#### 共用權限

- 將試算表分享給 `ADMIN_EMAILS` 中的帳號，並給予「**編輯者**」權限
- 一般員工登入的 Google 帳號，也需要至少「**編輯者**」權限才能寫入每日紀錄
  （試算表「共用」設定 → 加入帳號或設為「知道連結的使用者皆可編輯」）

## 二、部署到網路（讓手機可以開啟）

由於是純前端網頁，必須先放到一個有 `https://` 網址的靜態網站託管服務，員工才能用手機開啟。

### 方法 A：GitHub Pages（推薦，網址固定）

1. 在 GitHub 建立一個新的 repository（公開）
2. 用網頁介面「Add file → Upload files」上傳 `index.html`
3. 進入 repo 的 **Settings → Pages**，Source 選擇 `main` 分支、`/ (root)`，儲存
4. 約 1 分鐘後即可取得網址：`https://<帳號>.github.io/<repo名稱>/`
5. 之後若要更新內容，重新上傳 `index.html` 覆蓋即可，約 1 分鐘後自動重新部署

### 方法 B：Netlify（拖曳上傳，更快速）

1. 登入 https://app.netlify.com/（可用 Google 帳號登入）
2. 將 `index.html` 拖曳到部署區域
3. 取得網址：`https://xxxx.netlify.app`（可在 Site configuration → Change site name 改成好記的名稱）
4. 之後若要更新內容，到該 Site 的「Deploys」頁籤重新拖曳上傳即可

> ⚠️ 部署完成後，務必回到 Google Cloud Console 的 OAuth 用戶端設定，
> 把實際的網址（僅網域）加入「**已授權的 JavaScript 來源**」，否則會出現授權錯誤。

## 三、常見問題排除

### 「初始化失敗：讀取資料失敗 (403)：The caller does not have permission」
登入的 Google 帳號對該試算表沒有存取權限。請到試算表「共用」設定，將該帳號加入並給予編輯者權限。

### 畫面卡在「正在請求 Google 帳號授權...」一直轉圈圈
瀏覽器封鎖了頁面載入時自動彈出的授權視窗。系統會在約 6 秒後自動停止轉圈，
並改顯示「登入 Google」按鈕，請手動點擊該按鈕（手動點擊較不會被瀏覽器封鎖）。

### 「已封鎖存取權：授權錯誤」
- 確認網站網址（網域）已加入 OAuth 用戶端的「已授權的 JavaScript 來源」
- 確認 OAuth 同意畫面狀態：若為「測試中」，需將使用者帳號加入「測試使用者」名單
- 確認 OAuth 同意畫面已包含 `.../auth/spreadsheets` 與 `.../auth/userinfo.email` 兩個範圍

### 手機打不開網頁
- 確認是用 Chrome / Safari 等一般瀏覽器開啟，**不要用 LINE、Messenger、Instagram 等 App 內建瀏覽器**
  （這類內嵌瀏覽器會被 Google 封鎖 OAuth 登入）
- 確認開啟的是部署後的 `https://` 網址，而不是本機的檔案路徑

### 「系統管理」分頁顯示「您沒有管理員權限」
登入的 Google 帳號 Email 不在設定檔的 `ADMIN_EMAILS` 清單內。
請確認清單內容是否包含該帳號的完整 Gmail 地址（需與該帳號登入的 Google 帳號完全一致）。

### 新增員工 / 車牌後，「每日紀錄」分頁的下拉選單沒有更新
請確認上方「登入 Google」狀態顯示為「✅ 已登入 Google 帳號」，且試算表權限正確（見上方 403 問題）。
新增成功後，下拉選單會立即自動刷新。
